\section{Introduction}
\label{sec:intro}

We forecast next-24-hour electricity demand for each of the eight ISO New England (ISO-NE) load zones: \zone{ME}, \zone{NH}, \zone{VT}, \zone{CT}, \zone{RI}, \zone{SEMA}, \zone{WCMA}, \zone{NEMA\_BOST}. Inputs are a 24-hour window of weather rasters, demand, and calendar features. With $S=24$ history steps, $\nz=8$ zones, and 44-dimensional one-hot calendar features $C_t$, the model computes
\begin{equation}
    \hat Y_{t+1:t+24} \;=\; f_\theta\!\bigl(\, X_{t-S:t+24}\!\in\!\mathbb{R}^{(S+24)\times 7\times 450\times 449},\;\; Y_{t-S:t}\!\in\!\mathbb{R}^{S\times \nz},\;\; C_{t-S:t+24}\!\in\!\mathbb{R}^{(S+24)\times 44}\bigr),
    \label{eq:task}
\end{equation}
where $X$ are HRRR-style hourly weather rasters (7 channels) \citep{benjamin2016hrrr}. The metric is mean absolute percentage error (MAPE) on physical MWh values, averaged over the 24-hour horizon and the 8 zones:
\begin{equation}
    \mathrm{MAPE}(\hat Y, Y) \;=\; \frac{100}{24\,\nz}\sum_{h=1}^{24}\sum_{z=1}^{\nz}\frac{\bigl|\hat Y_{t+h,z}-Y_{t+h,z}\bigr|}{Y_{t+h,z}}.
    \label{eq:mape}
\end{equation}

\paragraph{Test-set caveat.} All MAPE numbers in this report come from our own self-evaluation harness on the \textbf{last 2 days of 2022} (2022-12-30 and 2022-12-31). This mirrors the cluster's \texttt{inference/test\_run.sh} default. The TA's official grader runs the submitted \texttt{model.py} on a held-out \textbf{2024} slice. Absolute numbers will differ; the model is data-year agnostic, so per-zone gaps and architectural trends should carry over.

\paragraph{Contributions.} (i)~A 1.75\,M-parameter CNN-Transformer baseline at 5.24\,\% test MAPE (\S\ref{sec:method}); (ii)~an honest negative result on a 2.29\,M-parameter encoder-decoder variant, with three diagnosed causes including a reproducible learning-rate scheduler reset bug (\S\ref{sec:archsearch}); and (iii)~a geographically-grounded attention extraction pipeline with four orientation-correctness checks built in (\S\ref{sec:attn}).

\section{Related Work}
\label{sec:relwork}

\paragraph{Patch-token Transformers for spatiotemporal data.} ViT \citep{dosovitskiy2021vit} introduced the ``image $\to$ patch tokens $\to$ Transformer'' recipe that templates our weather encoder. We swap ViT's linear patch embedding for a small CNN trunk because our $450{\times}449{\times}7$ raster is too coarse for direct patch tokenization. TimeSformer \citep{bertasius2021timesformer} extends ViT to video; Earthformer \citep{gao2022earthformer} specializes it to gridded earth-system data with cuboid attention. Both motivate our choice to share spatial position embeddings across timesteps and let attention mix the temporal dimension freely.

\paragraph{Tabular time-series Transformers.} PatchTST \citep{nie2023patchtst} shows that channel-independent patch tokens with a vanilla Transformer encoder beat heavily-engineered baselines on long-horizon forecasting. iTransformer \citep{liu2024itransformer} inverts the dimensions and tokenizes whole variables. Our tabular branch follows PatchTST: each timestep's $(\nz + 44)$-dim demand-plus-calendar vector is linearly embedded into one token, and attention mixes temporal context. The Temporal Fusion Transformer \citep{lim2021tft} is our reference point for the Part~2 encoder-decoder split.

\paragraph{Weather-aware foundation models.} ClimaX \citep{nguyen2023climax} shows that a single Transformer pretrained on multi-source climate fields transfers to downstream forecasting. We read it as evidence that joint weather + tabular Transformer encoders are the right inductive bias for our problem, even at our modest compute scale.

\paragraph{Normalization pipeline.} Every model and evaluator uses the same four-step chain, so cross-architecture comparisons are fair:
\begin{enumerate}
  \item \textbf{Compute statistics once} from 500 random training samples: per-channel z-score for weather $(\mu_x, \sigma_x \in \mathbb{R}^{7})$ and per-zone z-score for demand $(\mu_y, \sigma_y \in \mathbb{R}^{\nz})$. Cached in \texttt{runs/<model>/norm\_stats.pt}.
  \item \textbf{Train-time normalization.} The dataset's \texttt{\_\_getitem\_\_} z-scores both inputs and targets:
    \begin{equation}
        \tilde X = (X-\mu_x)/\sigma_x,\qquad \tilde Y = (Y-\mu_y)/\sigma_y.
        \label{eq:zscore}
    \end{equation}
    The loss is $\mathrm{MSE}(\hat{\tilde Y},\tilde Y)$, which keeps gradient magnitudes stable across zones with very different load levels.
  \item \textbf{Validation MAPE in physical space.} \texttt{compute\_mape(preds, targets, norm\_stats)} denormalizes both with $Y = \sigma_y\,\tilde Y + \mu_y$ before applying Eq.~\eqref{eq:mape}. Printed numbers are real MWh percentages.
  \item \textbf{Eval-wrapper closure.} At TA-grade time, the wrapper reads cached stats from the checkpoint, normalizes the grader's raw inputs, runs forward, denormalizes, and returns predictions in MWh.
\end{enumerate}
This chain is byte-for-byte identical across the Part~1 baseline, the Part~2 v1 encoder-decoder, and the Part~3 attention extractor.

\section{Method --- Part 1: CNN-Transformer Patch Baseline (40\,pts)}
\label{sec:method}

\subsection{Architecture}

The baseline is a hybrid CNN-Transformer (Figure~\ref{fig:arch}). A small CNN downsamples each $450{\times}449{\times}7$ weather snapshot into an $8{\times}8$ spatial-token grid ($P=64$). One tabular token per timestep encodes demand and calendar features. The result is a single sequence of length $(S+24)\cdot(P+1) = 48 \cdot 65 = 3120$ tokens.

The CNN trunk is a five-stage residual stack \citep{he2016resnet} of strided convolutions ($32 \to 64 \to 128 \to 128 \to D$) followed by adaptive average pooling to the target $G{\times}G$ grid. Each timestep produces $P$ spatial tokens that share a learnable spatial position embedding. The tabular token at history steps is a linear embedding of $[Y_t \,\Vert\, C_t]$; at future steps we replace the unknown demand with a learnable mask vector $\mathbf{m}\!\in\!\mathbb{R}^{\nz}$ (zeros at init) and embed $[\mathbf{m}\,\Vert\, C_t]$. Spatial and tabular tokens at the same timestep share a learnable temporal position embedding:
\begin{equation}
    \mathrm{seq}_{t,k} \;=\; \begin{cases} \mathrm{CNN}(X_t)_{k} + \mathbf{e}_{k}^{\mathrm{sp}} + \mathbf{e}_{t}^{\mathrm{tm}} & k\in[0,P) \\ W_{\mathrm{tab}}\,[Y_t\,\Vert\,C_t] + \mathbf{e}^{\mathrm{tab}} + \mathbf{e}_{t}^{\mathrm{tm}} & k=P\;\text{(tabular)}. \end{cases}
    \label{eq:assembly}
\end{equation}
A 4-layer pre-norm Transformer encoder with multi-head self-attention \citep{vaswani2017attention,ba2016layernorm} processes the sequence. We slice out the 24 future-tabular states and pass them through an MLP head to produce $\hat Y_{t+1:t+24}\!\in\!\mathbb{R}^{24\times \nz}$.

\subsection{Training}

We train with AdamW \citep{loshchilov2019adamw} at base learning rate $10^{-3}$, cosine annealing to zero \citep{loshchilov2017sgdr}, weight decay $10^{-4}$, batch size 4 per GPU, and dropout 0.1. The full configuration is in Table~\ref{tab:hparams} (Appendix~\ref{app:hparams}). The baseline trains continuously for 14 epochs on a single A100, in one SLURM job with no chained resumes, in $\sim$22\,h.

\subsection{Test results}

Table~\ref{tab:baseline} reports per-zone test MAPE on the last two days of 2022. Overall MAPE is \textbf{5.24\,\%}. The model is strongest on \zone{ME} (2.31\,\%) and \zone{NH} (3.69\,\%) --- the most rural, least demand-volatile zones --- and weakest on \zone{CT} and \zone{NEMA\_BOST}, the urban-coastal zones with the most fine-grained weather sensitivity.

\section{Architecture Search --- Part 2: Encoder-Decoder Variant (30\,pts)}
\label{sec:archsearch}

\subsection{Design rationale}

Inspired by TFT \citep{lim2021tft} and PatchTST \citep{nie2023patchtst}, we split the model into an \emph{encoder} over history and a \emph{decoder} over the 24 future-hour query tokens, connected by cross-attention. The hope: a clean encoder/decoder split improves fine-grained temporal localization on the urban-coastal zones where the baseline is weakest.

The decoder uses self-attention plus cross-attention into the encoder memory:
\begin{equation}
    \mathrm{Dec}(Q) \;=\; \mathrm{MHA}(Q, K_{\mathrm{mem}}, V_{\mathrm{mem}}),\qquad
    K_{\mathrm{mem}}, V_{\mathrm{mem}}\in\bigl\{\,\text{history-spatial},\;\text{future-spatial (v2 only)}\,\bigr\}.
    \label{eq:xattn}
\end{equation}
The two variants we trained share an identical joint encoder; the only architectural delta is \emph{which} subset of encoder memory the decoder reads. Figure~\ref{fig:v1v2} shows them side by side. v1 cross-attends to history-spatial keys only. v2 enables \texttt{use\_future\_weather\_xattn=True} and adds a second cross-attention path to future-spatial keys.

\subsection{Three honest reasons for v1's underperformance}
\label{sec:archsearch-reasons}

\paragraph{(1) Information disadvantage.} The v1 decoder is by default disconnected from future weather spatial tokens (left panel of Figure~\ref{fig:v1v2}). The encoder computes them, but only history-spatial keys reach the decoder. The Part~1 baseline's joint encoder, in contrast, sees future weather directly via positional embeddings on the unified sequence. This alone plausibly explains most of the gap on the urban-coastal zones, where future-weather timing is load-bearing.

\paragraph{(2) Reproducible LR-scheduler reset bug.} v1 ran as a 3-job SLURM chain with \texttt{--dependency=afterany}, each 24\,h. The per-epoch learning rates in the consolidated training log show:

\end{tabular}
\end{table}

Root cause: \texttt{training/train.py} saves \texttt{model.state\_dict()} and \texttt{optimizer.state\_dict()} but \emph{not} \texttt{scheduler.state\_dict()}. On resume, \texttt{CosineAnnealingLR(optimizer, T\_max=args.epochs)} re-initializes from scratch and the LR jumps back to $10^{-3}$, knocking the model out of its converged minimum. The baseline ran continuously for 14 epochs with proper cosine decay and hit its big-drop epoch (10.45\,\% $\to$ 7.64\,\% at epoch 9, Figure~\ref{fig:baseline-curves}) at small LR; v1 never gets there. Figure~\ref{fig:bvsv1} makes the bug obvious: v1 train loss flat-lines at $\sim$0.35 from epoch~7 onward, while the baseline train loss steadily falls to 0.20 by epoch~13. The fix is one line: save and restore \texttt{scheduler.state\_dict()} alongside the optimizer.

\section{Attention-Map Diagnosis --- Part 3, Track A (30\,pts)}
\label{sec:attn}

\subsection{Approach}

The Part~1 baseline uses \texttt{nn.MultiheadAttention} over the flat 3120-token sequence. Each layer $\ell$ produces an attention matrix $A^{(\ell)}\in\mathbb{R}^{B\times h\times 3120\times 3120}$, extracted via \texttt{need\_weights=True, average\_attn\_weights=False}. We keep only the rows for future-tabular queries and columns for history-spatial keys:
\begin{equation}
\begin{aligned}
    \mathcal{Q}_{\mathrm{fut}} &= \bigl\{\,t\cdot(P{+}1) + P : t\in[S, S{+}24)\bigr\},\quad |\mathcal{Q}_{\mathrm{fut}}| = 24, \\
    \mathcal{K}_{\mathrm{hist}} &= \bigl\{\,t\cdot(P{+}1) + p : t\in[0, S),\,p\in[0, P)\bigr\},\quad |\mathcal{K}_{\mathrm{hist}}| = S\cdot P = 1536, \\
    A^{(\ell)}_{\mathrm{slice}} &= A^{(\ell)}[\,:\,, \,:\,, \,\mathcal{Q}_{\mathrm{fut}},\, \mathcal{K}_{\mathrm{hist}}\,] \;\in\;\mathbb{R}^{B\times h\times 24\times 1536}.
\end{aligned}
\label{eq:slice}
\end{equation}
For every future hour $h_{\mathrm{f}}\in[1,24]$, $A^{(\ell)}_{\mathrm{slice}}$ gives a distribution over $24 \times 64$ pairs that answers: \emph{which historical map cells, at which historical hour, does the model rely on to predict hour $t{+}h_{\mathrm{f}}$?}

\subsection{Geographic orientation correctness}

We reshape the 64-cell spatial dimension to an $8{\times}8$ grid \emph{row-major}, matching \texttt{WeatherCNN.forward}'s \texttt{flatten(2)} layout: cell index $i$ maps to \texttt{(row=$i\div 8$, col=$i\bmod 8$)}. With \texttt{matplotlib.imshow(\,\dots, origin='upper')} this puts row~0 (north) on top and col~0 (west) on the left --- standard map orientation. As a falsifiable sanity check, the \zone{NEMA\_BOST} zone (eastern Massachusetts / metro Boston) should attend more to the eastern half of the grid than the western half:
\begin{equation}
    \sum_{r=0}^{7}\sum_{c=4}^{7} A^{\mathrm{NEMA\_BOST}}_{rc} \;>\; \sum_{r=0}^{7}\sum_{c=0}^{3} A^{\mathrm{NEMA\_BOST}}_{rc}.
    \label{eq:sanity}
\end{equation}
This assertion lives directly in \texttt{scripts/attention\_maps.py}. If it fails, the orientation is flipped and we fix in code before publishing figures (see Appendix~\ref{app:extraction}).

\subsection{Status and analytical questions}

The extraction script is implemented and verified locally on a CPU smoke test with synthetic input. The four target figures (\texttt{attention\_aggregate}, \texttt{attention\_per\_hour}, \texttt{attention\_extreme\_vs\_mild}, \texttt{attention\_per\_zone}) are produced by \texttt{scripts/attention\_maps.slurm} on Tufts HPC. Rendering needs the cluster's weather data and is the only outstanding piece after the 5/1 submission. Questions the figures will answer:
\begin{enumerate}
  \item \textbf{Which regions drive demand?} Hypothesis: a thick band along the Boston--Hartford corridor (concentrated load and weather variance).
  \item \textbf{Does the model track incoming weather?} Hypothesis: per-hour attention shifts west as forecast hour grows. New England weather typically advects west$\to$east, so the model should attend to upstream conditions that arrive at load centers later.
  \item \textbf{Do zones attend to different regions?} Hypothesis: \zone{ME}~$\to$ north; \zone{NEMA\_BOST}~$\to$ near-coast east; \zone{CT}~$\to$ south-west; \zone{WCMA}~$\to$ west.
\end{enumerate}
The NEMA\_BOST sanity check (Eq.~\eqref{eq:sanity}) is the load-bearing falsification test. If it passes, the qualitative claims above can be inspected directly from the four figures.

\section{Discussion}
\label{sec:discussion}

\paragraph{What worked.} The 1.75\,M-parameter baseline reaches 5.24\,\% test MAPE in 22\,h on a single A100, with no architecture search and no curriculum. The four-step normalization chain (\S\ref{sec:data}) was the biggest piece of engineering payoff: MAPE in physical MWh space plus uniform z-scoring gives apples-to-apples cross-architecture comparisons and a stable training signal across zones with very different load levels.

\paragraph{What didn't.} v1 lost to the baseline by 1.58\,pp despite a 30\,\% larger parameter count. The loss is concentrated on the urban-coastal zones (\zone{CT}, \zone{NEMA\_BOST}, \zone{RI}) where future-weather timing matters most --- consistent with the information-disadvantage hypothesis. The LR-scheduler reset bug (\S\ref{sec:archsearch-reasons}) is the dominant confound: v1's loss flat-lines at exactly the epoch where the baseline started its big drop. Until v2 finishes, we cannot separate ``architecture'' from ``information'' as the cause of v1's gap.

\paragraph{Limitations.} (i)~All numbers are on the last 2 days of 2022, a 2-window slice; small-sample variance is non-negligible and the TA's 2024 grade may differ. (ii)~The CNN trunk is a fixed 5-stage residual stack; we did not explore the spatial-encoder design space. (iii)~The attention extraction in \S\ref{sec:attn} is read-only: it diagnoses the trained model but does not feed back into a training-time geographic prior.

\section{Conclusion}
\label{sec:conclusion}

We presented a 1.75\,M-parameter CNN-Transformer that reaches 5.24\,\% test MAPE on the ISO-NE day-ahead demand task. The model tokenizes weather rasters with a small CNN trunk and concatenates them with calendar+demand tabular tokens into a unified Transformer-friendly sequence. A 2.29\,M encoder-decoder variant lost to the baseline. The dominant cause is a reproducible scheduler-reset bug in our chained-resume training infrastructure, not an architectural failure mode. The Part~3 attention extraction pipeline is implemented and orientation-checked; figures will land in the live presentation after one HPC sbatch.

\paragraph{Future work.} (i) Patch \texttt{scheduler.state\_dict()} save/load in \texttt{train.py}. (ii) Re-run v1 in a single uninterrupted SLURM window. (iii) Compare v1 vs.\ v2 to disambiguate ``architecture vs.\ information.'' (iv) A real-time HuggingFace Spaces demo using ISO Express + HRRR live data (skeleton in \texttt{space/}).

\section{Attention Extraction Procedure}
\label{app:extraction}

The four orientation-correctness checks in \texttt{scripts/attention\_maps.py}:

\begin{enumerate}
  \item \textbf{Slice direction}: rows of the sliced attention matrix are future-tabular query positions $\mathcal{Q}_{\mathrm{fut}}$; columns are history-spatial key positions $\mathcal{K}_{\mathrm{hist}}$ (Eq.~\eqref{eq:slice}).
  \item \textbf{Row-major reshape}: the 64-cell column dimension is reshaped to $(8, 8)$ row-major, matching \texttt{WeatherCNN.forward}'s \texttt{x.flatten(2)} layout (cell index $i$ $\to$ (row=$i\div 8$, col=$i\bmod 8$)).
  \item \textbf{Compass orientation}: \texttt{matplotlib.imshow(..., origin='upper')} so row 0 (north) is at the top, col 0 (west) is on the left.
  \item \textbf{NEMA\_BOST sanity assert}: Eq.~\eqref{eq:sanity}; if violated, code aborts before publishing figures.
\end{enumerate}

The pseudocode for the slice operation (Python):

\begin{lstlisting}[language=Python]
P, S = 64, 24                                  # spatial tokens / history len
fut_q   = [t * (P + 1) + P for t in range(S, S + 24)]    # 24 indices
hist_k  = [t * (P + 1) + p for t in range(S) for p in range(P)]  # 1536
A       = layer_attn[:, :, fut_q][:, :, :, hist_k]       # (B, h, 24, 1536)
A_8x8   = A.reshape(B, h, 24, S, 8, 8)                   # row-major

east = A_8x8[..., 4:].sum(dim=(-1, -2, -3))    # zone-conditioned later
west = A_8x8[..., :4].sum(dim=(-1, -2, -3))
assert east > west, "Orientation flipped"
\end{lstlisting}
